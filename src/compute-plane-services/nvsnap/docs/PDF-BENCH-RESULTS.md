## Per-workload notes

[^g]: gpt-oss-120b snapshot is rootfs-only (multi-GPU per CLAUDE.md
      rule 20 — no CRIU + cuda-checkpoint). Watcher detected the existing
      capture hash on the warm pod and reused it in ~1s (would be ~120s
      on a fresh capture). Restore uses per-pod OverlayFS over the
      captured tree (#194): lower stays RO + sharable, writes (vLLM
      torch_compile_cache, HF refs/main) land in upper. vLLM loaded the
      AOT compile cache from lower (4.79s compile vs ~30-60s/worker
      recompile that would happen with the old RO bind). Inference
      verified post-restore: POST /v1/completions returned "Paris" for
      "The capital of France is".

[^w]: **Routed via rootfs+OverlayFS path (agent v0.0.46, 2026-06-08).** CRIU +
      cuda-checkpoint was the original target for whisper-NIM but fails
      post-restore: Riva ASR pipeline aborts on cudaHostUnregister of the
      pinned host logits buffer (EINVAL — cuda-checkpoint does not serialize
      the host-pinned memory registration list), Triton's stub-process
      destructor double-throws, __cxa_call_terminate, abort.
      Trace: `cudaError_t 1 returned from cudaHostUnregister(pinned_host_logits_buffer_.data())`
      at `riva/asr/pipeline/asr_ensemble/streaming_asr_ensemble.cc:243`.
      Root cause sits in cuda-checkpoint
      (cui/dmal/rm/rm_checkpoint_mem.c:482/497 — save/restore should
      serialize+replay the host-registration list). Tracked as nvsnap#89.

      Workaround: rootfs+OverlayFS captures the warmed model cache
      (`/opt/nim/.cache` emptyDir, ~1.46 GiB of whisper-large-v3-fp16) and
      injects it as an overlay on a fresh restore pod. No CRIU process
      restoration, no host-pinned-memory replay needed — Riva starts clean
      against the warmed cache. The agent autoroutes Riva-backed NIMs to
      this path via `internal/agent/nim_backend.go::DetectBackend`; on
      `POST /v1/checkpoint` the agent returns HTTP 422
      `{"redirect":"rootfs","backend":"riva"}` and scripts handle the
      relabel automatically. Snapshot (5s) = blobstore upload time, not a
      CRIU dump. Restore (47s) = container startup + Riva server init
      against the pre-warmed model cache. Inference verified post-restore:
      `/v1/health/ready` returns `{"status":"ready"}` and `/v1/metadata`
      returns the correct `whisper-large-v3:h100x1-ofl-26.01.2-fp16` model
      profile. See nvsnap!73.

[^ds4]: Restore validated 2026-06-08 on agent v0.0.40 (init-container
        mount-prep + cache-root overlay coalescing + HostPathDirectoryOrCreate;
        nvsnap#202). **52% faster than cold (10m30s vs 22m02s)** — see
        the [DeepSeek-V4-Flash deep-dive](#deep-dive-deepseek-v4-flash-sglang-tp8)
        below for the phase-by-phase trace, predicted restore floor analysis,
        and design priorities.

## Deep-dive: DeepSeek-V4-Flash sglang TP=8

**Cluster**: GCP-H100-a (GKE asia-southeast1-c)
**Node**: gke-GCP-H100-a-GCP-H100-a-h100-pool-de45e9d3-5vwb (one of 8 H100 80GB)
**Workload**: DeepSeek-V4-Flash (671B fp8) on sglang v0.5.12.post1, TP=8
**Cold trace**: agent v0.0.36 (post-MR !66 whole-rootfs-restore), 2026-06-07 UTC
**Restore validation**: agent v0.0.40 (nvsnap#202), 2026-06-08

### Goal

Identify where time goes in the 22-min cold path so we know which
phases can be skipped via nvsnap's rootfs-restore and which are floor.

### Method

- Truly cold start (host hf-cache wiped via node-debug pod across all
  8 H100 nodes; kernel page cache dropped)
- Sampler running every 30s recording: pod phase, hf-cache size,
  /root/.cache size, md0 write rate, latest sglang log line
- Source yaml: `deploy/k8s/benchmarks/deepseek-v4-flash.yaml` with
  `nvsnap.io/capture=true` so the watcher snapshots automatically
- Single run, T0 = 2026-06-07T03:22:12 UTC

### Cold path: 22m02s broken down

| T+ (mm:ss) | Phase | Duration | Disk write | Notes |
|------------|-------|---------:|----------:|-------|
| 0:00 → 3:50 | Container + nvsnap-init + Python imports + nccl init | 3m50s | 0–244 MB/s burst | nvsnap-init writes 887 MB to /root/.cache (sitecustomize + libnvsnap); kubelet pod setup; sglang starts |
| 3:50 → 11:17 | **Model download (149 GB)** | **~7 min** | 360 MB/s avg, 442 MB/s peak | Direct from HF Hub (we are in nvsnap-system, not the NVCF workload namespace, so the proxy cache does NOT apply). GCE 10 Gbps + HF Hub multi-stream range fetch from Cloudflare R2. |
| 11:17 → ~14:30 | Weight load disk → VRAM + MHC prewarm | ~3 min | idle | sglang mmaps safetensors, copies to GPUs. MHC prewarm `path=attn` ~7.9s per rank. |
| 14:54 → 16:12 | **DeepGEMM JIT Pre-Compile session** | **~1m18s** | idle | Sglang's "typically 10-20 mins" warning was misleading for this workload — actual was ~80s. |
| 16:12 → 17:00 | DeepGEMM warmup | ~10s | idle | Final warmup pass after pre-compile session |
| 17:00 → 22:02 | **CUDA graph capture (36 batch sizes)** | **~5 min** | trickle | Batches bs=256 → bs=1, capturing one cudagraph per shape. Initial `bs=256` alone takes ~6m23s reported in tqdm but appears amortized via TP. |
| 22:02 | Application startup complete, /v1/models 200 OK | | | **Ready** |

### What's reachable by nvsnap's rootfs-restore

| Phase | Cold time | Skippable via rootfs? | How |
|-------|----------:|------------------------|-----|
| Container + python init | 4 min | **No** | Fresh process, fresh K8s pod; CRIU multi-GPU is blocked |
| Model download | 7 min | **Yes** | hf-cache captured into `tree/volumes/hf-cache/`; restored pod mounts it RO at `/var/lib/hf-cache-bench` — sglang "finds local snapshot" instantly |
| Weight load → VRAM | 3 min | **No** | VRAM goes cold on pod restart; mmap+H2D must run again |
| DeepGEMM Pre-Compile | 1.3 min | **Yes (partial)** | `/root/.cache/deep_gemm/` captured (5 entries, 0.7 MB); covers most kernels |
| CUDA graph capture | 5 min | **No** | CUDA graphs reference VRAM allocator state; cannot be persisted without `cudaCheckpoint` (which is single-GPU only per NVIDIA upstream) |

**Predicted restore floor: ~13 min (vs 22 min cold) — 40% reduction.**

Breakdown:
- 4 min container + python (mandatory)
- 0 min model download (skipped via mount)
- 3 min weight load (mandatory)
- 0 min DeepGEMM pre-compile (skipped via cache)
- 5 min CUDA graph capture (mandatory, NVIDIA-blocked)
- 1 min startup misc

### Restore validation (result)

Captured hash: `7eb6a326fb36b6d6eb1281e8cddbe4f793d6eac38794ed1766b44125f45a772e`
Manifest has 357 mount points (272 python-dist-packages, 45 rootfs-extract,
32 sglang-workspace, 5 deep-gemm-cache, 1 torch-cache, 2 captured volumes).

Validated 2026-06-08 on agent v0.0.40 (nvsnap#202 init-container
mount-prep + cache-root overlay coalescing + HostPathDirectoryOrCreate).

| Phase | Cold | Restore | Saved |
|-------|-----:|--------:|------:|
| Container + nvsnap-init + python + NCCL init | 3m50s | ~2m | ~2m |
| Model download (149 GB) | ~7m | 0 (mounted) | ~7m |
| Weight load disk → VRAM + MHC prewarm | ~3m | ~2m | ~1m |
| DeepGEMM JIT pre-compile | ~1m18s | covered by captured cache | ~1m |
| CUDA graph capture (36 batch sizes) | ~5m | 6m25s | -1.5m (capture-completeness gap) |
| **Total** | **22m02s** | **10m30s** | **11m32s (52%)** |

Beat the predicted floor by ~2.5 min — captured DeepGEMM/TileLang/tvm-ffi
caches covered enough shape variants that restore-side JIT was bounded.
Cuda graph capture is the remaining floor (NVIDIA cuda-checkpoint is
single-GPU only — closed upstream).

**Admission overhead on the 357-mount restore (the nvsnap#202 promise):**
- `kubectl apply` (webhook patches only, no kernel work): 2.91 s
- Async agent prep (16 workers): 78 ms for 311 overlay mounts after
  cache-root coalescing
- Pod created → main container Running: 2 s

### Design priorities (what this trace tells us)

1. **whole-rootfs-restore (MR !66) is right-sized for this workload.**
   The ~7 min download saving + ~1.3 min JIT saving = ~8.3 min off cold
   restore. Code is correct; this bench validates it.
2. **CUDA graph capture is the floor.** Without NVIDIA cuda-checkpoint
   multi-GPU support (closed permanently per upstream), we can't go
   below ~13 min for this workload. Engine-side "release-CUDA, dump
   CPU-only" via CRIU was discussed and rejected — same complexity as
   per-engine catalog maintenance for marginal gains.
3. **Capture timing matters less than feared.** Today's run captured at
   Ready+60s and the captured tree has all the warm caches (deep_gemm,
   flashinfer, tvm-ffi, sgl-workspace). Earlier "captured-too-early"
   failure was actually because a previous capture cycle reused stale
   manifest via content-addressed hash collision — not a true
   Ready+60s problem.
4. **Disk is NOT the bottleneck.** md0 (RAID-0 NVMe) sustained
   360 MB/s during download (peak 442) and dropped to <2 MB/s during
   engine init. Anywhere disk IO is fast, the only remaining cost is
   network + compute.
5. **Network is also not a bottleneck on GCE.** Even without unbound
   proxy (we're in nvsnap-system), HF Hub direct delivers 360 MB/s
   sustained over the 10 Gbps node NIC.
